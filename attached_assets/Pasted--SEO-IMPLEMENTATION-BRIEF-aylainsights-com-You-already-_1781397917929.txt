# SEO IMPLEMENTATION BRIEF — aylainsights.com

You already have the current aylainsights.com build. Before making ANY changes,
complete Step 0 below. Do not blindly overwrite existing work — audit first,
then reconcile this spec against what's already there.

Ayla Insights is a pre-launch B2B SaaS selling an AI front-office copilot to
DENTAL PRACTICE OWNERS AND MANAGERS (not to patients). All SEO targets the
practice-buyer, never patients searching for a dentist.

## STEP 0 — AUDIT THE EXISTING SITE FIRST (do this before changing anything)

Inventory what currently exists and report it back before implementing:

1. List every existing route/page and its current <title>, meta description,
   and H1.
2. Note which pages from the target architecture (below) already exist, which
   are missing, and which exist under a different URL.
3. Flag any existing copy, title tag, meta description, alt text, or schema
   that claims "HIPAA compliant" or "SOC 2 certified" — these are compliance
   violations and must be corrected regardless of anything else.
4. Note existing schema markup, sitemap.xml, robots.txt, and analytics/Search
   Console setup — present or absent.
5. Identify any existing blog/resources content.

Then produce a short GAP REPORT with three buckets:
   - ALREADY DONE (matches this spec — leave alone)
   - NEEDS EDIT (exists but wrong/incomplete — fix to match spec)
   - MISSING (build new per spec)

Implement only NEEDS EDIT and MISSING items. Preserve anything already correct.
Where the existing copy is good but not SEO-optimized, refine it — do not
discard the founder's voice or rewrite working pages from scratch. Where this
spec conflicts with an intentional existing design choice, flag it for the
founder rather than silently overriding.

## GLOBAL RULES (apply everywhere)

1. COMPLIANCE GUARDRAIL — never use the phrase "HIPAA compliant" anywhere in
   copy, title tags, meta descriptions, alt text, or schema. Only "built for
   HIPAA" or "designed for HIPAA." This is a legal requirement. For security
   keywords use "secure dental software" / "dental data security" instead.
2. Brand pairing — every <title> pairs the brand with "dental" context.
   Format: "Page Topic | Ayla Insights — AI Front Office for Dental Practices"
3. One H1 per page, matching that page's primary keyword.
4. Title tags ≤60 chars where possible; meta descriptions ≤155 chars.
5. Descriptive, keyword-aware alt text on all meaningful images (especially
   dashboard screenshots).
6. Semantic HTML: proper h1→h2→h3 hierarchy, not styled divs.
7. Mobile-first and fast — practice managers browse on phones. Lazy-load
   images, compress assets, target Core Web Vitals green.
8. Every page has a canonical tag. No duplicate-content pages.

## SITE ARCHITECTURE (one page per keyword cluster)

Target routes (reconcile against existing routes in Step 0):
- /                     Home
- /features             Features overview (sections per pillar)
- /dentrix              Dedicated Dentrix integration page (high-intent)
- /security             Security & data protection
- /about                Founder + mission
- /contact  or /demo    Book a discovery call (primary CTA destination)
- /resources            Blog/resources hub
- /resources/[slug]     Individual posts

Primary CTA sitewide: "Book a discovery call." Secondary: "Join the waitlist."
No public pricing page, no self-serve signup (pre-launch posture). If a pricing
page currently exists, flag it in the gap report — do not publish pricing.
